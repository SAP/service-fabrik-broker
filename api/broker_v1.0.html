Service Fabrik Broker API - v1.0

Welcome to the Service Fabrik Broker API documentation.

Most of the below listed operations are service-specific and thus the Service Fabrik Broker will interact with the corresponding agents of the service instance in order to delegate the request.

HTTP Headers ¶

If not stated otherwise, the HTTP header must set

Content-Type: application/json

for all requests and responses.

Errors ¶

The API uses standard HTTP codes to indicate that errors occurred during the request. Each related response body contains a description of the error within JSON format:

{
  "code": 400,
  "message": "An error occurred because of bad parameters."
}

Special errors are:

  • 501: indicating that the agents configured for the service instance’s service plan have not implemented the requested operation at all

  • 502: indicating that the agents configured for the service instance’s service plan have implemented the requested operation, but currently no agent is eligible to accept it

Security ¶

All endpoints except /info are secured via OAuth2 access tokens (provided by the Cloud Foundry UAA Server of the CF system the Service Fabrik is registered to). You need appropriate privileges for the Cloud Foundry service instance you want to perform an operation on (i.e., having the SpaceDeveloper in the service instance’s space). In order to authenticate yourself, you need to obtain a token from the UAA and send it in the HTTP header of the call to the broker:

$ ACCESS_TOKEN=$(curl -s -X POST -d "grant_type=password&client_id=cf&client_secret=&response_type=token&username=YOUR-USERNAME&password=YOUR-PASSWORD" "http://uaa.bosh-lite.com/oauth/token" | jq -r .access_token)
$ curl -H "Authorization: Bearer $ACCESS_TOKEN" "https://service-fabrik-broker/v1/api/service_instances/SERVICE-INSTANCE-GUID?plan_id=SERVICE-PLAN-ID" | jq .
{
  "operational": true
}

URI Templates ¶

In this documentation, we express our URIs according to https://tools.ietf.org/html/rfc6570.

General ¶

Retrieve information about the broker.

Show Broker Information
GET/info

Response
  200 - success

In case of success, a JSON object is returned which contains the broker name, the version of the implemented API and an indicator which shows whether the broker is ready to perform all its implemented operations (i.e., if all bootstrapping operations have been completed).

Example

{
  "name": "service-fabrik-broker",
  "api_version": "1.0",
  "ready": true
}

      "type": "string",
      "description": "Service Broker Name"
    },
    "api_version": {
      "type": "string",

    },
    "ready": {

State ¶

Retrieve the current state of a service instance.

Show State
GET/service_instances/{instance_guid}{?plan_id}

The state of a service instance is described by an indicator showing whether it is operational.

URI Parameters
instance_guid
string (required) 

the guid of the service instance

plan_id
string (optional) 

the unique service plan id

Response
  200 - success
  401 - unauthorized
  403 - access to resource forbidden

In case of success, a JSON object is returned which contains information about the operational status, and optionally additional information. The Service Fabrik obtains these data from the corresponding agent(s) running on the service instance.

Example

{
  "operational": true,
  "details": {
    "number_of_files": 0
  }
}

Schema
HideShow

{
  "$schema": "http://json-schema.org/draft-04/schema#",
  "type": "object",
  "required": [
    "operational"
  ],
  "properties": {
    "operational": {
      "type": "boolean",
      "description": "True if the service instance is available and False otherwise."
    },
    "details": {
      "description": "Any additional information returned from the service agent."
    }
  }
}

Backup ¶

Backup service instance data safely.

Start Backup
POST/service_instances/{instance_guid}/backup

Endpoint for triggering a manual backup on a BOSH service instance.

URI Parameters
instance_guid
string (required) 

the guid of the service instance

Request

The request body must be a JSON object containing the type of the backup (online/offline), and may optionally contain the space_guid and the unique service plan id. A backup of type offline may involve downtimes of the service instance itself, whereby an online backup does not cause any downtime.

Body Parameters
type
string (required) 

the backup type (either online or offline)

space_guid
string (optional) 

the space guid of the service instance

plan_id
string (optional) 

the unique service plan id

Example

{
  "type": "online",
  "space_guid": "11111111-1111-1111-1111-111111111111",
  "plan_id": "22222222-2222-2222-2222-222222222222"
}

Schema
HideShow

{
  "$schema": "http://json-schema.org/draft-04/schema#",
  "type": "object",
  "properties": {
    "type": {
      "type": "string",
      "enum": [
        "online",
        "offline"
      ],
      "description": "Type of the backup"
    },
    "space_guid": {
      "type": "string",
      "description": "Space guid of the service instance"
    },
    "plan_id": {
      "type": "string",
      "description": "Unique service plan id of the service instance"
    }
  }
}
Response
  202 - success
  401 - unauthorized
  409 - another operation is already in progress for the service instance

In case of success, the backup request was accepted and started. The response body will be a JSON object which contains the name of the operation and the computed backup guid. As the backup operation is asynchronous, its state can be checked by polling the GET /service_instances/{instance_guid}/backup endpoint.

Example

{
  "name": "backup",
  "guid": "a6b39499-8b8b-4e1b-aaec-b2bc11d396e4"
}

Schema
HideShow

{
  "$schema": "http://json-schema.org/draft-04/schema#",
  "type": "object",
  "properties": {
    "name": {
      "type": "string",
      "description": "Name of the operation"
    },
    "guid": {
      "type": "string",
      "description": "GUID of the backup"
    }
  }
}

Endpoint for triggering the abortion of a running backup on a BOSH service instance.

URI Parameters
instance_guid
string (required) 

the guid of the service instance

space_guid
string (optional) 

the space guid of the service instance

Response
  200 - success (currently no backup in progress for this service instance)
  202 - success
  401 - unauthorized

In case of success, the abortion request was accepted. The cleanup of resources may require some time and usually won’t complete immediately. The current state can be checked by polling the GET /service_instances/{instance_guid}/backup endpoint. The response body will always be an empty JSON object.

Scheduling Backups ¶

Schedule Backup
PUT/service_instances/{instance_guid}/schedule_backup

By default when a service key is generated for a service instance (either via bind service or create service key methods by CF), if the instance is not already setup for automated backups then service fabrik broker automatically schedules backup with a default schedule interval of ‘daily’. The schedule times are randomized in the next 24 hours to achieve some randomization in backup. This default schedule interval of ‘daily’, can be overridden by definining ‘backup_interval’ at service level in the services manifest. [Refer PostgreSQL manifest for example]

The below API can be used to Update the automated backup schedule interval as defined by service fabrik broker at the service instance key generation time or Create the backup schedule even before service fabrik broker creates backup schedule.

URI Parameters
instance_guid
string (required) 

the guid of the service instance

Request

The request body must be a JSON object containing the type of the backup (online/offline), repeat interval and may optionally contain the timezone to be taken into consideration for job runs. A backup of type offline may involve downtimes of the service instance itself, whereby an online backup does not cause any downtime.

Body Parameters
type
string (required) 

the backup type (either online or offline)

repeatInterval
string (required) 

repeat interval of the backup. Supported formats: valid cron expression or human-readable intervals (ex. 4 hours, 6 hours 30 minutes,etc.)

timeZone
string (optional) 

timezone to consider when running the batch job. If timezone is not specified, then default timezone of UTC would be considered

Example

{
  "type": "online",
  "repeatInterval": "*/30 */6 * * *",
  "timeZone": "America/New_York"
}
